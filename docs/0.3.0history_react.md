> **版本节点**: v0.3.0（ReAct 稳定性专项）

## 一、本次分析做了什么

本次对话从架构层面对 Rooster 的执行稳定性进行了系统性诊断，完整阅读了以下核心文件：

| 文件 | 行数 | 关键发现 |
|------|------|---------|
| `src/agents/router.py` | 483 | `[DIRECT]` 和 `[REFRAME]` 都走 MissionRunner，路由策略有误 |
| `src/agents/runners/mission_runner.py` | 903 | REMAND 重试失忆；Checkpoint 完整实现但默认关闭 |
| `src/agents/executor.py` | 1054 | ReAct 循环写得很好，是 Claude Code 风格，不需要重写 |
| `src/agents/strategist.py` | 455 | `plan_stream()` 无超时保护，是最高风险点 |
| `src/prompts/executor.md` | 198 | 强制要求 FINAL_REPORT JSON 输出，SoloRunner 不需要 |

## 二、架构现状快照

### 实际调用链（2026-05-27 现状）

```
用户输入
  → [LLM #1] Router._triage_via_llm()
  → [LLM #2] Reframer.reframe()           （[REFRAME] 任务）
  → [LLM #3] Strategist.plan_stream()     （无超时！max_tokens=32768）
  → [LLM #4] AgentExecutor.run() × N      （每个 SubTask 独立实例）
  → [LLM #5] Auditor.review()             （仅叶节点）
  → [LLM #6] AgentExecutor 重试           （REMAND，但失忆重跑）
```

**简单任务最少 5 次串行 LLM 调用。**

### 路由策略（现状）

```
[TALK]     → SoloRunner  ✅ 正确
[DIRECT]   → MissionRunner  ❌ 应走 SoloRunner
[REFRAME]  → MissionRunner  ✅ 正确（真正需要多步规划）
[SCHEDULE] → _handle_schedule()  ✅ 正确
[BLOCK]    → 拦截  ✅ 正确
```

## 三、发现的 7 个根因

### 🔴 致命（必须立即修复）

**根因 1：`plan_stream()` 无超时保护**
- 位置：`src/agents/strategist.py` L175
- 问题：`chat_stream()` 外层没有 `asyncio.timeout()`，LLM 流中途断开 → Rooster 永久挂死
- 对比：`plan()` 方法有超时，但 MissionRunner 调用的是 `plan_stream()`

**根因 2：REMAND 重试 Executor 失忆**
- 位置：`src/agents/runners/mission_runner.py` L451-457
- 问题：重试时只注入 `previous_audit_cmd`（审计指令文字），完全不包含上次工具调用的返回值
- 效果：重试的 Executor 不知道上次做了什么，往往重蹈覆辙

### 🟠 严重（Phase 1 修复）

**根因 3：`[DIRECT]` 走错路由**
- 位置：`src/agents/router.py` L192
- 问题：清晰任务仍走 MissionRunner（Strategist + Auditor），多 3 次 LLM 调用

**根因 4：Strategist JSON 流式解析脆弱**
- 位置：`src/agents/strategist.py` L213-268
- 问题：手写 regex + 深度计数器实时解析 JSON，格式轻微偏移 → FAILSAFE 降级

### 🟡 中等（Phase 2-3 修复）

**根因 5：Checkpoint 默认关闭**
- 位置：`mission_runner.py` L101，`.env` 中 `CHECKPOINT_ENABLED=false`
- 问题：长任务崩溃必须从零重来；基础设施完整，改一行配置即可

**根因 6：dep_results 截断过激**
- 位置：`mission_runner.py` L307
- 问题：子任务间只传 2000 字符，复杂搜索结果被截断，下游选错候选

**根因 7：Executor 强制 JSON 输出**
- 位置：`src/prompts/executor.md`
- 问题：SoloRunner 模式不需要 FINAL_REPORT JSON，多一层格式化负担

## 四、核心结论

> **`executor.py` 的 ReAct 循环本身已经是 Claude Code 风格，写得很完整。**  
> **问题不在 Executor，而在它外面套了 3-4 层编排，只有 20% 的任务才能高效进入它。**

### 与 Claude Code 的对比

| 维度 | Claude Code | Rooster v0.3.0 |
|------|------------|---------------|
| 单任务 LLM 调用次数 | 2次（含执行循环） | 5-6次 |
| Context 连续性 | 单一连续 context | 每个角色断一次 |
| 工具失败重试 | 原地推理，同一 context | REMAND + 失忆重试 |
| 规划方式 | chain-of-thought 内联 | 独立 Strategist + JSON |
| 长任务续跑 | 有限 | 已实现，默认关闭 |
| **多 agent 真正并行** | ❌ 不支持 | ✅ 支持（核心优势） |
